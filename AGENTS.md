# ZLK 開発ガイド

このファイルは，リポジトリ全体で作業するエージェントが従うべき方針と手順を定める．

## 用語

- **開発者**：エージェントに指示を出し，このプロジェクトを遂行する者を指す．
- **特段の理由のない限り**：その方針から逸脱する理由や，逸脱によって得られる明確なメリットがない場合は，逸脱を禁止するという意味である．逸脱する合理的な理由がある場合は，理由，影響，および得られるメリットを開発者へ報告する．

## プロジェクト概要

ZLK は，実験的な関数型プログラミング言語である．本リポジトリでは，ZLK のコンパイラを開発する．

言語およびコンパイラの主な機能：

- Hindley–Milner 型推論
- 代数的データ型
- 閉じた構造的レコード型
- パターンマッチ
- PEG パーサー
- クロージャ変換
- JVM バイトコード生成
- 線形型（実装中）

## 開発方針

- 最適化よりも正しさの優先
- 各コンパイラフェーズの責務と境界の明確化
- 各フェーズにおける入力 IR と出力 IR の明示
- 後段フェーズの都合や前提の前段フェーズへの持ち込み禁止
- 構文解析後の AST の変更禁止
- 既存の抽象化を迂回しない，局所的かつ必要最小限の変更

## アーキテクチャ

コンパイラフェーズ，IR，パッケージの責務，および用語については，`ARCHITECTURE.md`を参照すること．

### コンパイラパイプライン

「依存先」列には，そのフェーズが利用する先行フェーズの番号を示す．

| # | フェーズ | 依存先 |

| 2 | `Parser` | 1 |
| 3 | `NameEvaluator` | 2 |
| 4 | `ConstraintExtractor` | 3 |

`ARCHITECTURE.md`のコンパイルフェーズ図とこの一覧は同じパイプラインを表す．フェーズを追加，削除，または変更する場合は，両者の一貫性を確認する．

## テスト構成

テストコード：`src/test/java/zlk`

- `ConstrainerTest.java`：型制約抽出
- `ReconTest.java`：型再構築
- `PatternMatchTest.java`：パターンマッチ検査
- `ParseErrorTest.java`：構文エラー処理
- `FeatureTest.java`：言語機能と実行時の意味論
- `tester`：コンパイルフェーズごとのテスト支援ユーティリティ

## ビルドとテスト

前提環境：

- JDK 23

リポジトリのルートで次を実行する．

```sh
./mvnw test
```

特定のテストクラスだけを実行する場合：

```sh
./mvnw -Dtest=ReconTest test
```

ただしWindows環境ではそれぞれ

```cmd
mvnw.cmd test
mvnw.cmd -Dtest=ReconTest test
```

すべての不具合修正には，その不具合を再現する回帰テストを追加する．テストの配置は，変更対象に最も近い既存クラスを優先する．

- 構文解析・構文エラー：`ParseErrorTest`
- 型制約抽出：`ConstrainerTest`
- 型再構築・型推論：`ReconTest`
- パターンマッチ：`PatternMatchTest`
- 言語機能・実行時の意味論：`FeatureTest`

変更後は，対象テストに加えて原則として `./mvnw test` を実行する．Windows 環境では `mvnw.cmd test` を実行する．

## 文書の表記

### 句読点と箇条書き

- 日本語の句読点として「，」と「．」の使用
- 箇条書きの項目における，可能な限りの体言止め
- 体言止めへの変換が不自然または困難な場合における，項目の分割，分類，または構成の再検討
- 意味の明確さを損なう場合に限った，体言止めによらない記述

### 文体

- 開発者との対話における敬体の使用
- ソースコードのコメントや開発者向けドキュメントなど，開発者および実装者に向けた日本語テキストにおける常体の使用
- 診断メッセージ，README，利用者向けドキュメントなど，コンパイラの利用者に向けた日本語テキストにおける敬体の使用
- 対象読者が曖昧な場合における，文体の決定前の開発者への確認

## コーディング規約

- `ARCHITECTURE.md`に定めたプロジェクト共通の省略形の使用，および新しい省略形を導入する場合における，既存コードとの整合性の確認と必要に応じた同文書の表への追加
- 不変オブジェクトの優先
- 選択肢が閉じたインターフェースにおける `sealed interface` の優先
- 可能な限り網羅的な `switch` の記述
- リフレクションの回避
- 隠れた副作用の回避
- 型推論ロジックの重複禁止
- `Type` 抽象化の迂回禁止
- 正しさの確認前における最適化の禁止
- 特段の理由のない限り，命名，インデント，コメントの詳細さ，記述方法，設計上の慣例を既存のコードに合わせた，周囲のコードに自然に馴染む実装

## 変更時の手順

1. 関連するコンパイラフェーズと IR の特定
2. 対象フェーズの入力IR，出力IR，変更する情報の所有者，および後続フェーズが依存する不変条件の確認
3. ASTを変更せずに情報を受け渡す方法の検討
4. 既存の実装とテストの読解，および期待される振る舞いの確認
5. 不具合修正における，可能な限り先行した失敗する回帰テストの追加
6. フェーズ境界を崩さない，必要最小限の変更
7. 対象フェーズに最も近いテストの実行
8. `./mvnw test`（Windows環境では`mvnw.cmd test`）によるコンパイラ全体への回帰の有無の確認
9. 変更箇所の`ARCHITECTURE.md`との整合性を確認

## エージェント向けルール

- 開発者との対話における，特段の理由のない限りの日本語および敬体の使用
- ソースコードのコメントや開発者向けドキュメントなど，開発者および実装者に向けた日本語テキストにおける，特段の理由のない限りの日本語および常体の使用
- 診断メッセージ，README，利用者向けドキュメントなど，コンパイラの利用者に向けた日本語テキストにおける，特段の理由のない限りの日本語および敬体の使用
- 既存コードの命名規則，外部 API の名称，一般に英語で記述する技術用語などに対する，日本語使用規則の適用除外
- 指示が計画やアイデアの提示に留まるか，コード編集やコマンド実行まで含むか曖昧な場合における，無断での編集・実行の禁止と開発者への作業範囲の確認
- 調査を伴わないファイル構成，API，テスト結果の推測禁止
- コード変更前における関連実装とテストの確認
- 明示的な依頼を伴わない，開発者の既存の変更の削除・上書き禁止
- 完了報告における変更内容と実際に実行した検証の簡潔な提示

## Git 操作

- 開発者から対象操作について明確な指示がない場合における，`git commit` と `git rebase` の実行禁止
- `git commit` または `git rebase` の指示の有効範囲を，指示を受けた直後に行う一回の操作に限定
- 同じ作業中に再度 `git commit` または `git rebase` が必要になった場合における，開発者からの明確な再承認の取得
- 承認された操作の前提や対象が変わった場合における，承認の流用禁止，変更内容の説明，および再承認の取得
